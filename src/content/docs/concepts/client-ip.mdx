Arcjet needs the public IP address of incoming requests for much of what it does.
By default,
Arcjet uses the IP as a `characteristic` to differentiate users.
If that doesn’t work correctly,
several good users may be denied access because they cannot be differentiated
from a bad user.
Or a malicious user could get around rate limits by appearing as different
users.
Arcjet also uses the client IP for geolocation and other metadata.

Finding the correct client IP address can be tricky.
How to find that IP depends on the framework you build with,
the platform you deploy to,
and whether there are proxies or similar services in front of it.

When you are developing your app locally,
there is no public IP.
When testing Arcjet,
you may want to overwrite the IP.

## Framework

Each Arcjet SDK is made for a particular framework and _should_ detect the client IP
correctly from incoming requests out of the box.
Some frameworks provide that IP address on the request directly
(such as `request.ip` and `request.requestContext.identity.sourceIp`),
in other cases it can be inferred from headers
(see the next section “[Platform][section-platform]”).

If no IP is detected in production,
you’ll see a big warning in the logs.
In this case,
if you do know where to find the correct IP address,
you can [overwrite the client IP][section-overwrite-ip].
But please also [open an issue][arcjet-arcjet-js-issues] so that we can fix this bug.

## Platform

Arcjet looks at request headers when a framework does not provide the trusted
client IP directly.
Whether to trust these addresses depends on the platform your app is running on.
Arcjet infers whether an app is running on Fly, Render, or Vercel
(but not Cloudflare) from the environment.
On other platforms Arcjet looks for common headers.

| Platform   | Header                                                                                                                                                                                             |
| ---------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Cloudflare | [`cf-connecting-ipv6`][cloudflare-cf-connecting-ipv6], [`cf-connecting-ip`][cloudflare-cf-connecting-ip]                                                                                           |
| Fly        | [`fly-client-ip`][fly-client-ip]                                                                                                                                                                   |
| Render     | `true-client-ip`                                                                                                                                                                                   |
| Vercel     | [`x-forwarded-for`][vercel-x-forwarded-for], [`x-real-ip`][vercel-x-real-ip], [`x-vercel-forwarded-for`][vercel-x-vercel-forwarded-for]                                                            |
| Other      | `do-connecting-ip`, `fastly-client-ip`, `forwarded-for`, `forwarded`, `true-client-ip`, `x-appengine-user-ip`, `x-client-ip`, `x-cluster-client-ip`, `x-forwarded-for`, `x-forwarded`, `x-real-ip` |

If you use several platforms,
such as Cloudflare in front of Vercel,
you can pass `platforms` so that the correct headers will be looked at.

## Proxies

When you put services like proxies or load balancers in front of your
application,
the IP addresses that Arcjet finds based on the framework and headers,
will be that of the last proxy.

Well-behaving services will use particular headers (typically `x-forwarded-for`)
to forward the client IP address that connects to them.
But because malicious users can also set that header,
and because proxies could potentially be bypassed,
Arcjet cannot blindly trust such values.

The safe way to handle this is with an allowlist of trusted IP ranges.
When you put a service like Cloudflare in front of your application,
and know that it sets headers correctly,
and know which IP addresses belong to that service,
you can configure Arcjet to skip its IP addresses.

## Overwriting client IPs

It is possible to overwrite the client IP address.
To make this easy locally but secure when deployed,
how to do this is different in development and production.

### Development

In development you can pass an `x-arcjet-xxx` header.
This value is preferred over any other detected IP address.
An example with Curl is:

```sh
curl https://example.com --header "x-arcjet-xxx: 76.76.21.21" --verbose
```

This header is ignored in production because otherwise malicious users could
set it.
But in development it is useful to simulate different client IP addresses
without changing code and restarting your app,
and to check how that affects Arcjet behavior,
particularly around rate limiting and IP geolocation.

```ts
import arcjetBun, { slidingWindow } from "@arcjet/bun";
import { env } from "bun";

const arcjet = arcjetBun({
  key: env.ARCJET_KEY!,
  // To illustrate, allow 3 requests per minute per IP address.
  rules: [slidingWindow({ interval: 60, max: 3, mode: "LIVE" })],
});

export default {
  fetch: arcjet.handler(async function (request) {
    request.ip = request.headers.get("x-some-header-that-you-trust");

    const decision = await arcjet.protect(request);

    if (decision.isDenied()) {
      return new Response("Access Denied", { status: 403 });
    }

    return new Response("Hello world");
  }),
  port: 3000,
};
```
