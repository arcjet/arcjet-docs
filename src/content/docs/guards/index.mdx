Guards apply Arcjet security rules **inside AI agent tool calls** and anywhere else you process untrusted input. Pass inputs directly, get a decision back. No `Request` object needed.

They are for the places HTTP middleware can't reach: inside tool handlers, between agent steps, in queue consumers, and across agentic pipelines that fan out across many services.

## The problem

AI agents call tools. Tools receive untrusted input. Your HTTP middleware never sees any of it — guards run inside the tool handler, before input reaches the model.

| Problem | Solution |
|---|---|
| **Injected instructions** — a `fetch` tool or external API returns content containing instructions that re-enter the model context unchecked. | **Prompt injection detection** — Arcjet scans tool inputs _and results_ for jailbreaks, instruction overrides, and role-play escapes before they reach the model. Evaluated in the cloud so detection stays up to date without redeploying. |
| **Sensitive data exposure** — a user pastes a credit card number or phone number into a prompt that gets logged or sent verbatim to a third-party model provider. | **Sensitive information detection** — Arcjet detects PII — email addresses, phone numbers, credit card numbers, and more — locally via WebAssembly. The raw text never leaves your server; only the classification result does. |
| **Budget exhaustion** — a user floods the agent with requests, burning through your LLM token quota or per-user allowances. | **Rate limiting** — token bucket, fixed window, and sliding window algorithms. Use token bucket to model LLM token budgets directly — deduct estimated tokens before each model call and deny when the bucket is empty. |

### 2. Paste this prompt into your coding agent

```txt
First, confirm the Arcjet MCP server is connected and you are authenticated (use the MCP tool to verify — if not connected, stop and ask me to connect it before continuing).

Next, explore the current project:
- Identify the language and framework (JavaScript/TypeScript or Python, and which framework)
- Find where untrusted input enters the system: AI agent tool handlers, chat message handlers, queue consumers, webhook handlers, or similar
- Note any existing rate limiting, input validation, or security middleware already in place

Then propose to me what to protect with Arcjet Guards and how:
- If JavaScript/TypeScript: use @arcjet/guard (npm install @arcjet/guard)
- If Python: use arcjet.guard (pip install arcjet)
- If the language is not supported yet, say so and stop
- List the specific files and functions you plan to change, and which guard rules make sense (prompt injection, rate limiting, sensitive information, or a combination)
- Ask me to confirm before making any changes

Once I confirm, implement the guards:
- Configure rules at module scope (not inside the handler)
- Start all new rules in DRY_RUN mode
- Add guard calls at the entry point of each handler before any other logic
- Log denied decisions with has_error() / hasError() rather than throwing immediately
- Use the Arcjet MCP server to check for any integration errors and resolve them
- When the integration looks correct, ask me if I want to switch from DRY_RUN to enforcement mode
```
