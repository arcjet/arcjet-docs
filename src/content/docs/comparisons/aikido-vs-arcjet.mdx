Arcjet is an alternative to [Aikido](https://www.aikido.dev/zen) Zen Firewall
for protecting your applications from bots, implementing rate limiting, and
defending against other attacks. Both products offer security features, but
there are some key differences to consider when choosing between them.

## Key differences: Aikido vs Arcjet

1. **Security components for developers**: Arcjet provides a range of security
   components that developers can integrate directly into their applications,
   including
   <Link.Page href="/bot-protection/quick-start">bot management</Link.Page>,
   <Link.Page href="/rate-limiting/quick-start">rate limiting</Link.Page>,
   <Link.Page href="/email-validation/quick-start">email verification</Link.Page>,
   <Link.Page href="/sensitive-info/quick-start">personal information detection & redaction</Link.Page>,
   a full
   <Link.Page href="/shield/quick-start">web application firewall (WAF)</Link.Page>
   that covers SQL injection, cross-site scripting, file inclusion attacks, and
   others, and <Link.Page href="/filters">filter rules</Link.Page> for
   expression-based traffic control. Arcjet also provides
   <Link.Page href="/ai-protection">AI Runtime Protection</Link.Page>: bot
   detection and prompt injection protection for AI endpoints, per-user token
   budget enforcement, and PII detection in AI request/response flows, plus
   <Link.Page href="/guards">Arcjet Guards</Link.Page>, which applies the same
   primitives inside AI agent tool handlers, queue consumers, and multi-step
   pipelines where there's no HTTP request. Aikido's
   Zen Firewall focuses on runtime application protection: blocking dangerous
   user input (injection, path traversal, SSRF, etc.), per-route and per-user
   rate limiting, and blocking bots, Tor traffic, known threat actors, and
   traffic by country. Aikido also offers a broader cloud security platform
   (code scanning, cloud scanning, etc.) that is not directly comparable to
   Arcjet's SDK.
2. **Integration**: Arcjet is designed to integrate as a library directly
   into your codebase. You define rules in code and call Arcjet alongside
   your application logic, and it does not modify your runtime or
   frameworks. Arcjet works with any modern JS/TS framework or bundler, and
   any Python application (supporting both async and sync APIs) and rules
   can be adjusted dynamically at runtime as regular code. Aikido's Zen
   Firewall for JS only works with Node.js and [monkey-patches your
   code](https://github.com/AikidoSec/firewall-node/blob/2f39052390e6ceff845a8f3addfefe7237b42f8d/instrumentation-wasm/src/js_transformer/helpers/insert_code.rs#L12)
   [at
   runtime](https://github.com/AikidoSec/firewall-node/blob/2f39052390e6ceff845a8f3addfefe7237b42f8d/library/agent/Agent.ts#L515)
   to insert security checks, which can lead to unexpected behavior,
   latency issues, and makes debugging more difficult. Configuration for
   Aikido is handled primarily through environment variables and the
   Aikido dashboard rather than rules-as-code inside your application.
   Arcjet does support dashboard- and MCP-managed
   <Link.Page href="/remote-rules">Remote Rules</Link.Page> for operational
   changes (rate limit, bot, filter, and Shield rules without a redeploy),
   but rules-as-code remains the default configuration model — not an
   afterthought to a dashboard-first product.
3. **Framework, runtime & language support:** Arcjet currently focuses on the
   JavaScript, TypeScript, and Python ecosystems, with native SDKs for JS
   platforms like Next.js, Node.js, Bun, React Router, SvelteKit, NestJS, and Python
   FastAPI and Flask (async and sync APIs), and other web servers. Aikido's Zen
   Firewall has a wider language surface area, with agents for Node.js, Python,
   PHP, Java, .NET, and Ruby. However, Aikido's monkey-patching approach
   causes issues with certain frameworks, such as [not being able to use
   custom rate limits by user ID in
   Next.js](https://github.com/AikidoSec/firewall-node/blob/main/docs/next.md)
   (because the agent's `setUser` function isn't supported in Next.js's
   bundling), Next.js requiring `output: "standalone"`, and [problems with
   bundlers](https://github.com/AikidoSec/firewall-node/blob/main/docs/bundler.md).
   ESM is supported with a `--require @aikidosec/firewall/instrument`
   preload, but [Zen cannot protect ESM
   sub-dependencies](https://github.com/AikidoSec/firewall-node/blob/main/docs/esm.md)
   of an ESM package — the V8 engine doesn't let Node observe the
   evaluation of inner ESM packages.
4. **Real time analysis:** Aikido uses a "local-only" approach to analyzing
   requests, meaning all decisions are made based on data available locally.
   Aikido handles bot detection [through user agent
   analysis](https://help.aikido.dev/zen-firewall/miscellaneous/bot-protection-details),
   with optional configuration for blocking specific IPs or countries and
   periodic refreshes of threat data. [Arcjet takes a "local-first"
   approach](https://thenewstack.io/arcjet-brings-ai-security-analysis-local-into-your-code/)
   where analysis is attempted first locally and if a high confidence decision
   can't be made, combines the local analysis with cloud-based intelligence to
   provide more accurate and up-to-date threat detection. Arcjet's cloud
   component allows it to leverage global threat data, IP reputation data,
   machine learning models, and other advanced techniques whilst still keeping
   sensitive data local. As a result, Arcjet can provide more effective
   protection against new threats with rule updates as they emerge.
5. **Advanced client signals for bot detection:** Arcjet provides
   <Link.Page href="/bot-protection/advanced-signals">advanced bot signals</Link.Page>,
   a browser-based detection layer that runs a WebAssembly module on the
   client to collect environment anomalies — headless markers, automation
   framework fingerprints, browser-quirk inconsistencies — and persists a
   continue token in the `aj_signals` cookie reused across navigations.
   Server-side `detectBot` evaluates the signals alongside user-agent
   classification, IP reputation, and an optional missing-cookie
   <Link.Page href="/filters">filter rule</Link.Page>
   (`len(http.request.cookie["aj_signals"]) eq 0`) in a single decision, so
   you can layer client-side analysis, server-side classification, and
   script-execution enforcement without integrating three products. Rules
   roll out under `LIVE` / `DRY_RUN` modes, and a signals failure surfaces
   as `ARCJET_SIGNALS` in `decision.reason` so you can branch your
   response. Aikido's bot detection remains focused on user-agent
   signature matching and IP-based controls, with no equivalent to
   client-side signal collection or expression-based filter rules.
6. **Agentic workloads and non-HTTP entry points:** Aikido's Zen Firewall
   instruments HTTP request handling and database/library call sites to catch
   injection-shaped attack patterns. It is built around the request-response
   model and has no surface for inspecting an LLM tool call, a queued job,
   or a step in a multi-agent pipeline. That blind spot is where most agent
   damage actually occurs: a model fetches a web page and re-enters its
   context loaded with hidden instructions; a tool call burns tokens in a
   runaway loop; a background job ships a support ticket to a third-party
   LLM with PII attached. <Link.Page href="/guards">Arcjet Guards</Link.Page>
   applies token-bucket rate limiting, prompt injection detection, and PII
   detection inline at the points an agent actually fans out: inside MCP
   tool handlers, between agent steps, in queue consumers, and across
   multi-service pipelines. You call `guard()` with the input directly; no
   `Request` object required. JS/TS via `@arcjet/guard`; Python via
   `arcjet`.

## When to choose which

If you're building a modern application in JavaScript, TypeScript, or Python and
care about predictable behavior, testability, and treating security as code,
**Arcjet is usually the better default**. If you need a multi-language runtime
firewall tightly coupled to Aikido's broader security platform, **Zen Firewall
can be the right fit**.

### Choose Arcjet if...

- **Your core stack is JavaScript, TypeScript and/or Python** You live in
  Next.js, Node, Bun, React Router, SvelteKit, etc. You want a modern SDK that behaves
  like any other library, works in serverless environments, and doesn't depend
  on runtime patching or agents.

- **You want security as part of your application design** You're happy to write
  rules in code, in the same place as your auth, billing, and routing logic. You
  want to express "block this bot", "rate limit per account", or "reject signups
  with disposable emails" as normal functions and ship them via your usual code
  review and CI.

- **You care about strong primitives beyond "firewall"** Bot protection, rate
  limiting, WAF, email verification, signup protection, PII detection/redaction,
  filter rules, AI Runtime Protection, and Guards are all part of the Arcjet
  product suite. You'd rather compose these into flows than sprinkle generic
  "block this request" calls around.

- **You're building AI applications** You need to protect AI endpoints from
  automated clients inflating token costs, detect prompt injection attacks, or
  enforce per-user token budgets. Arcjet's AI Runtime Protection covers abuse,
  budget control, and data loss prevention with the same SDK you use for
  everything else.

- **You're shipping an agent, MCP server, or background pipeline** Aikido's
  Zen Firewall is built around HTTP request handling; the risky surface in
  agentic code is what happens *after* the request enters your app — tool
  calls that re-enter the model with attacker-controlled content, runaway
  loops that burn tokens, and fan-out jobs that ship PII to third-party
  models. <Link.Page href="/guards">Arcjet Guards</Link.Page> runs the same
  primitives inline at those points.

- **You want hybrid local + cloud analysis, not just IP lists** You like the
  idea of a local WebAssembly engine plus cloud-side context (global reputation,
  richer analysis) with clear, explicit calls from your code — not a black-box
  agent making decisions you can't easily trace.

In short: if your product is primarily JavaScript, TypeScript, and/or Python and
you want security that feels like part of your application architecture, Arcjet
is the better, developer-first choice.

### Choose Aikido if...

- **You have a genuine polyglot estate** You're running Node, PHP, Java, .NET,
  Ruby, etc., and you want one vendor with agents for all of them, integrated
  into a broader platform that also does code, cloud and container scanning.

- **You want agent-style runtime protection with minimal code changes** You're
  comfortable installing agents that instrument frameworks, built-ins and
  libraries, and you prefer to drive most policy from a dashboard and
  environment variables rather than writing rules in code.

- **You're already bought into Aikido's platform** You want Zen as another
  module in the same place you manage SAST, cloud scanning and other security
  controls, and are happy with the agent's Node.js-only runtime patching
  (with the ESM sub-dependency limitation noted above).

Aikido makes more sense when the primary constraint is "one security vendor and
agent model across many languages" and you're happy to accept that operational
shape.

## FAQ

### Can I use Arcjet and Aikido at the same time?

Yes. Arcjet runs as a normal library inside your JavaScript, TypeScript or
Python code, while Aikido Zen runs as an agent that injects into your runtime.
You can start by adding Arcjet for things that are easier as rules-as-code (bot
protection, signup protection, rate limiting) while keeping Aikido for other
languages or existing policies. Over time you can migrate traffic and rules to
Arcjet route by route.

### What if I have services in PHP, Java, .NET or Ruby?

Arcjet is currently focused on JavaScript, TypeScript, and Python. If a large
part of your stack is in other languages, you have a couple of options:

- Use Arcjet for the supports languages and user-facing APIs where latency and
  developer ergonomics matter most.
- Keep Aikido on other languages, or use their platform where you don't want to
  touch application code.

If your primary product surface is JavaScript, TypeScript, and/or Python, you'll
usually get a better developer experience by standardizing on Arcjet there and
treating other languages as exceptions.

### Which should I choose for a greenfield SaaS?

If the app is primarily JavaScript, TypeScript and/or Python, and you control
the code, Arcjet is the better default:

- Fits naturally into handlers/middleware for all JavaScript, TypeScript, and
  Python frameworks and runtimes.
- Modern - ESM for JS and Python 3.10+ with support for sync and async APIs.
- Rules live in your codebase, reviewed and tested like everything else.
- Strong primitives for bots, rate limits, WAF, email, signup flows, and PII.

Aikido makes more sense only if you're optimizing for “one security vendor
across many languages” from day zero, which is rare for typical products.

### How do I tune false positives?

With Arcjet, tuning is done in code: you can log-only, branch on user, plan,
route, or environment, and add conditions around rules like any other business
logic. You can version and test these changes like normal code with conditionals
and your existing feature flag systems.

With Aikido, most tuning is done in the dashboard and environment variables.
That can be quick for small changes, but harder to keep in sync with how your
application actually behaves, especially across environments.
